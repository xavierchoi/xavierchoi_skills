# A/B Test Draft Generator

LinkedIn 포스트의 A/B 테스트용 초안 2개를 생성하는 프롬프트입니다. 두 버전은 반드시 다른 구조와 훅을 사용하여 차별화되어야 합니다.

---

## 생성 원칙

### 1. 필수 참조

다음 파일들을 반드시 참조하여 작성하세요:

- **문체 가이드**: `/Users/xavier/code/LinkedIn/data/style-guide.json`
- **작성 가이드**: `prompts/post-writer.md`

### 2. 차별화 전략

두 버전은 **반드시** 다른 접근 방식을 사용해야 합니다:

#### 버전 A: 논리적/구조적 접근

다음 중 하나를 선택:

| 구조 | 설명 | 예시 |
|------|------|------|
| **문제 → 해결 → 결과** | 독자의 pain point에서 시작하여 해결책과 성과 제시 | "매일 3시간씩 반복 작업에 쓰고 있었습니다 → Claude Code로 자동화 → 이제 30분이면 끝" |
| **질문형 훅** | 강렬한 질문으로 시작하여 답을 전개 | "코드 리뷰에 하루 몇 시간을 쓰시나요?" |
| **숫자/데이터 시작** | 구체적인 수치로 관심 유발 | "3일 걸리던 작업을 2시간으로 줄였습니다" |
| **Before/After** | 변화 전후를 대비하여 효과 강조 | "이전: 수동으로 100개 파일 검토 → 이후: 한 줄 명령어로 완료" |

**버전 A 특징:**
- 명확한 구조와 논리적 흐름
- 구체적인 단계나 포인트 제시
- 실용적이고 액션 가능한 내용
- 전문적이고 신뢰감 있는 톤

#### 버전 B: 감성적/서사적 접근

다음 중 하나를 선택:

| 구조 | 설명 | 예시 |
|------|------|------|
| **스토리텔링** | 개인 경험이나 상황으로 시작하여 인사이트 도출 | "어젯밤 새벽 2시, 배포 직전 버그를 발견했습니다..." |
| **강렬한 주장형** | 도발적이거나 예상을 뒤엎는 문장으로 시작 | "코드 리뷰는 시간 낭비입니다. (단, 이렇게 하면요)" |
| **공감 시작** | 독자가 겪는 어려움에 공감하며 시작 | "또 야근입니다. 반복되는 코드 수정 때문에..." |
| **반전/대비** | 예상과 다른 결론이나 발견으로 시작 | "AI가 개발자를 대체할 거라 생각했습니다. 틀렸습니다." |

**버전 B 특징:**
- 감정적 연결과 공감 유도
- 스토리나 경험 중심
- 호기심과 몰입 유발
- 개인적이고 친근한 톤

1. /review 명령어로 변경사항 한눈에 파악

## 피드백 반영 가이드

사용자가 피드백을 제공하면:

1. **버전 선택 시**: 선택된 버전을 기반으로 수정
2. **요소 조합 요청 시**: 두 버전의 장점을 결합
3. **구체적 수정 요청 시**: 해당 부분만 수정하고 나머지 유지
4. **톤/길이 조정 요청 시**: 문체 가이드 범위 내에서 조정

**무제한 iteration**: 사용자가 만족할 때까지 수정을 반복합니다.
